Frog Engine

A 100% object-oriented 3D game engine for Dolet. No ECS.

Vulkan rendering Scene graph glTF + PNG Alpha

Overview

Frog is a classic object-oriented 3D engine. Every object in the world is a Node in a scene graph, and you drive the whole thing through a small static Engine facade. The Vulkan backend, swapchain, and descriptor plumbing stay hidden — you work with cameras, models, transforms, and textures.

Engine

Static lifecycle: window, frames, meshes, textures, camera rendering.

Assets

Load models (glTF/glb), textures (PNG), audio, and shaders from a mounted root.

Scene

Node tree with Node3D, Camera3D, transforms, parenting, and collision flags.

Math

Vec3, Mat4, Quat, Color plus angle / rng / clamp helpers.

Bring everything into scope with a single import:

import frog

Project Setup

A Frog game is a normal Dolet program. You typically import the standard library, the engine, and your own game module:

import std
import frog
import game.runtime

Before loading any asset, mount a root directory. Everything you load later resolves relative to it. A common pattern handles both running from source and running next to the built executable:

asset_probe: str = PathOps.resolve_path("assets\\models\\cube.gltf")
if PathOps.exists(asset_probe):
    Assets.mount("assets")
else:
    Assets.mount(PathOps.resolve_from_exe("..\\assets"))

Quick Start

A complete minimal scene: a window, a ground plane, one model, and a camera.

import std
import frog

# 1. Configure the window before starting
Engine.window.create("My Game", 1280, 720)
Engine.properties.set_vsync(0)
Engine.properties.set_instance_culling(1)
Engine.debug.render_stats(1)

# 2. Mount assets and build a scene
Assets.mount("assets")
scene: Scene = Scene.create()

actor: Node3D = Node3D.create("Actor")
actor_model: Model3D = Assets.models.load("models/actor.gltf")
camera: Camera3D = Camera3D.create("Camera")
ground: Node3D = Node3D.create("Ground")

# 3. Configure nodes (chainable)
actor.position(0.0, 1.0, 0.0)
actor.set_model(actor_model)
camera.position(0.0, 2.0, 6.0)
camera.rotation(0.0 - 12.0, 180.0, 0.0)
camera.set_fov(82.0)
camera.make_current()
ground.as_static()
ground.with_box_collision()
ground.scale(800.0, 1.0, 800.0)

# 4. Build the tree and hand it to the engine
actor.add_child(camera)
scene.add_child(actor)
scene.add_child(ground)
Engine.scene.set(scene)
Method chaining. Node setters such as position, rotation, scale, and set_model return the node, so you can chain them: actor.position(0,1,0).scale(2,2,2).

The Game Loop

Frog does not hide the main loop. You own it. Engine.start() brings up Vulkan, begin_frame() returns the delta time in seconds, and a render call submits the frame for the current camera.

if Engine.start() == 0:
    return

Engine.capture_mouse()
while Engine.running() == 1:
    dt: f32 = Engine.begin_frame()
    # ... update your game state with dt ...
    Engine.render_camera(camera)

Engine.destroy_scene()
Engine.shutdown()
Wrap this in your own attach Engine: static fun run() to keep main tidy, then just call Engine.run().

Guide: Textures

A texture is a resource, so it loads through Assets and gets worn by a node — exactly like a model. One call reads the PNG and uploads it to the GPU; the node binds it. A TextureSpec describes how the image should appear.

# Load + GPU-upload model and texture (each happens once)
actor_model: Model3D = Assets.models.load("models/actor.gltf")
actor_tex:   Texture2D = Assets.textures.load("textures/actor.png", TextureSpec.smooth())

# The node wears both — no manual mesh/texture ids
actor: Node3D = Node3D.create("Actor")
actor.set_model(actor_model)
actor.set_texture(actor_tex)

Calling set_model and set_texture in either order is safe — the node binds the texture as soon as it has both. Each mesh keeps its own descriptor set, and meshes with no texture fall back to a 1×1 white pixel so geometry still renders.

TextureSpec — controlling appearance

A TextureSpec sets filtering, wrapping, and color space. Presets cover the common cases:

PresetUse for
TextureSpec.smooth()Default. Linear filtering, repeat wrap, sRGB color. Models and photos.
TextureSpec.pixel()Sharp nearest filtering + clamp. Pixel art and crisp sprites.
TextureSpec.ground()Linear + repeat + anisotropy. Tiled floors/walls seen at glancing angles.
Modifier / fieldEffect
.as_data()Mark as linear (non-color) data: normal/roughness maps.
.wrap(mode)Set U/V/W wrap: 0 REPEAT, 1 MIRROR, 2 CLAMP_EDGE, 3 CLAMP_BORDER.
mag_filter / min_filter0 = NEAREST (sharp), 1 = LINEAR (smooth).
anisotropy0 = off, else max anisotropy (e.g. 8.0 / 16.0).
srgb1 = color (gamma-corrected), 0 = linear data.
# Pixel-art variants of one model — different images, same mesh
red:  Texture2D = Assets.textures.load("textures/chicken_red.png", TextureSpec.pixel())
blue: Texture2D = Assets.textures.load("textures/chicken_blue.png", TextureSpec.pixel())
chicken_a.set_texture(red)
chicken_b.set_texture(blue)

# Tiled ground, no shimmer at distance
ground.set_texture(Assets.textures.load("textures/grass.png", TextureSpec.ground()))

# Normal map — linear data, not color
n: Texture2D = Assets.textures.load("textures/wall_normal.png", TextureSpec.smooth().as_data())
PNG support. Textures decode through a pure-Dolet PNG decoder: full zlib/DEFLATE inflate, all five row filters, color types 0/2/3/4/6, and bit depths 1/2/4/8/16.
Low-level path. If you work with raw mesh ids (e.g. GPU instances), Engine.upload_texture(path) / Engine.upload_texture_spec(path, spec) and Engine.bind_texture(mesh_id, tex_id) are still available.

Guide: Viewmodels (held items)

A first-person item should stay fixed relative to the camera, not the world. Rather than parenting it in the scene graph, render it by writing a raw model matrix built from the camera's world transform.

# Build: world = camera_world * local_offset, then push raw bytes
model: Mat4 = camera_world.mul(item_local)
buf: i64 = Memory.malloc_zeroed(64)
model.write_to_ptr(buf)
Engine.set_model_matrix_raw(item_mesh, buf)

Because set_model_matrix_raw bypasses the usual TRS pipeline, the item follows the camera exactly each frame and reads as a true held viewmodel.

Rig tip. A clean setup is a head node with the camera and the held item as its children, so look direction drives both together.

Guide: Building & Optimization

Compile a game to a native executable with the Dolet compiler. An optimization level can be passed directly.

doletc game.dlt -o game.exe -O2      # or -O0 / -O1 / -O3
FlagEffect
-O0No optimization (default). Fastest builds, best for debugging.
-O1Basic optimization.
-O2Recommended for release builds. Strong speed gains, stable.
-O3Most aggressive optimization.

The level is forwarded straight to the C backend, so an unoptimized -O0 build and an -O2 build can differ dramatically in frame rate. Use -O2 when benchmarking or shipping.
