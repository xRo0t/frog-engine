Frog Engine

A 100% object-oriented 3D game engine for Dolet. No ECS.

Vulkan rendering Scene graph glTF + PNG Alpha

Overview

Frog is a classic object-oriented 3D engine. Every object in the world is a Node in a scene graph, and you drive the whole thing through a small static Engine facade. The Vulkan backend, swapchain, and descriptor plumbing stay hidden; you work with scenes, cameras, meshes, materials, transforms, and textures.

Engine

Window, lifecycle, frame timing, rendering, GPU selection, and debug flags.

Scene

The active world. It owns a node tree, an active camera, and an environment.

Mesh3D

A visible node: transform, built-in shape, material, texture, and collision helpers.

Material3D

Reusable surface data: color, texture, alpha, roughness, metallic, and future shader data.

Assets

Load models, textures, audio, and shaders from a mounted game-owned root.

Math

Vec3, Mat4, Quat, Color plus angle / rng / clamp helpers.

Bring everything into scope with a single import:

import frog

Mental Model

Think in layers. The game owns code and assets. Frog owns reusable engine objects. The renderer stays behind the Engine facade unless you explicitly choose a low-level path.

LayerYou writePurpose
ApplicationEngine.window.create(...), Engine.start(), loopBoot and drive the native game executable.
WorldScene.create(), scene.set_environment(env)Own the active world, camera, and node tree.
ObjectsNode3D, Mesh3D, Camera3D, Raycast3DRepresent things that exist in the scene.
ResourcesMaterial3D, Texture2D, Model3D, Shader3DReusable data shared by one or many objects.
Low levelEngine.upload_model_mesh, Engine.register_instanceOptional path for high-count rendering and special systems.
Default path. New games should start with Scene + Mesh3D + Material3D. Drop to raw mesh ids only when you are building a performance system such as instancing, terrain chunks, particles, or a viewmodel.

Project Setup

A Frog game is a normal Dolet program. You typically import the standard library, the engine, and your own game module:

import std
import frog
import game.runtime

Before loading any asset, mount a root directory. This is required; Frog does not choose a default asset folder.

Assets.mount("assets")

Keep code and static files separate. The engine does not guess the root; the game chooses it.

my-game/
  src/main.dlt
  src/game/runtime.dlt
  assets/models/player/scene.gltf
  assets/textures/player.png

With the layout above, Assets.models.load("models/player/scene.gltf") resolves through the mounted assets directory. Loading before Assets.mount(...) returns an asset error instead of guessing a path.

Quick Start

A complete minimal scene with no external assets: one camera, one floor, one cube, and a render loop.

import std
import frog

# 1. Configure before Engine.start()
Engine.window.create("My Game", 1280, 720)
Engine.properties.set_present_mode(PresentMode.mailbox())
Engine.properties.set_target_fps(0)
Engine.properties.set_render_scale(1.0)
Engine.properties.set_gpu_preference(GpuPreference.auto())
Engine.properties.set_frustum_culling(1)
Engine.debug.render_stats(1)

# 2. Build the scene
scene: Scene = Scene.create()

env: Environment = Environment.create()
sky: Sky = Sky.create()
sky.set_zenith(Color.rgb(0.47, 0.63, 0.91))
sky.set_horizon(Color.rgb(0.72, 0.82, 0.96))
sky.set_lower(Color.rgb(0.80, 0.88, 1.0))
env.set_sky(sky)
env.set_fog(Fog.view_distance(sky.fog_color(), 240.0))
scene.set_environment(env)

camera: Camera3D = Camera3D.create()
camera.position(0.0, 3.0, 8.0)
camera.rotation(0.0 - 18.0, 180.0, 0.0)
camera.set_fov(82.0)
scene.add_child(camera)

floor_mat: Material3D = Material3D.create()
floor_mat.set_color(Color.rgb(0.18, 0.45, 0.22))
floor: Mesh3D = Mesh3D.create()
floor.set_mesh(Mesh3D.shape.plane())
floor.set_material(floor_mat)
floor.scale(12.0, 1.0, 12.0)
floor.as_static()
floor.collision_box(12.0, 0.25, 12.0)
scene.add_child(floor)

cube_mat: Material3D = Material3D.create()
cube_mat.set_color(Color.rgb(0.08, 0.62, 0.92))
cube: Mesh3D = Mesh3D.create()
cube.set_mesh(Mesh3D.shape.cube())
cube.set_material(cube_mat)
cube.position(0.0, 0.75, 0.0)
cube.scale(1.5, 1.5, 1.5)
cube.collision_box(1.0, 1.0, 1.0)
scene.add_child(cube)

Engine.scene.set(scene)

# 3. Start and render
if Engine.start() == 1:
    while Engine.running() == 1:
        dt: f32 = Engine.begin_frame()
        cube.rotate(0.0, 45.0 * dt, 0.0)
        Engine.render()

    Engine.destroy_scene()
    Engine.shutdown()
Order matters. Configure window/properties, create your scene, call Engine.scene.set(scene), then call Engine.start(). Load assets before the loop; do not load textures or rebuild materials every frame.

First Object

For a visible object, create a Mesh3D, choose a mesh shape, assign a material, set transform/collision, then add it to the scene.

Names are optional. Use Mesh3D.create() for quick objects. Use Mesh3D.create("Crate") when the name helps debugging, raycast interaction, logs, or editor tools.

When the object uses a texture or model file, mount assets once during setup:

Assets.mount("assets")
texture: Texture2D = Assets.textures.load("textures/crate.png", TextureSpec.pixel())

mat: Material3D = Material3D.create()
mat.set_color(Color.rgb(0.08, 0.62, 0.92))
mat.set_texture(texture)
mat.set_surface(0.0, 0.85)

cube: Mesh3D = Mesh3D.create("Crate")
cube.set_mesh(Mesh3D.shape.cube())
cube.set_material(mat)
cube.position(0.0, 0.75, 4.0)
cube.scale(1.5, 1.5, 1.5)
cube.as_static()
cube.collision_box(1.0, 1.0, 1.0)
scene.add_child(cube)
UseWhen
Mesh3DThe object is visible geometry in the scene.
Node3DThe object is only a transform, parent, trigger, controller, or gameplay node.
Material3DThe visual surface should be reusable or has more than one property.
cube.set_color(...)Fast shortcut for one-off debug/simple objects.

The Game Loop

Frog does not hide the main loop. You own it. Engine.start() brings up Vulkan, begin_frame() returns the delta time in seconds, and a render call submits the frame for the current camera.

if Engine.start() == 1:
    Engine.mouse.capture()
    while Engine.running() == 1:
        dt: f32 = Engine.begin_frame()
        # ... update your game state with dt ...
        Engine.render()

    Engine.destroy_scene()
    Engine.shutdown()
Wrap this in your own attach Engine: static fun run() to keep main tidy, then just call Engine.run().

PresentMode

ModeUse whenTradeoff
PresentMode.auto()You want the engine default behavior.Uses the legacy vsync preference to choose a supported mode.

To verify fallback behavior, read the requested mode before startup and the active mode after startup:

requested: PresentMode = Engine.properties.present_mode()
if Engine.start() == 0:
    return
active: PresentMode = Engine.properties.active_present_mode()
requested_name: str = PresentMode.name(requested)
active_name: str = PresentMode.name(active)
print(@"present requested={requested_name}, active={active_name}")

Collision visualization

Engine.debug.collisions(1)

# Custom boxes are per-frame; call after Engine.begin_frame().
Engine.debug.box(
    Vec3.make(0.0, 1.0, 0.0),
    Vec3.make(1.0, 2.0, 1.0),
    Color.rgba(0.1, 0.8, 1.0, 0.9)
)

The renderer discovers regular Node3D collision boxes and enabled Raycast3D nodes automatically, then submits the debug outlines as batched instanced line draws. If collision debugging is never enabled, Frog performs no physics-debug traversal, creates no debug GPU buffers, and submits no debug draw calls. Disabling it after first use stops traversal and rendering while retaining the cached GPU buffers for fast re-enabling.

Custom game nodes

For normal gameplay objects, start with Node3D and add behavior in your game modules. For high-count custom data, define a game-owned struct that keeps the base node fields first, then add game fields after them.

    name: str = ""

    kind: i32 = 0

Recommended: reusable material

Guide: Textures

A texture is a resource, so it loads through Assets and gets worn by a node — exactly like a model. One call reads the PNG and uploads it to the GPU; the node binds it. A TextureSpec describes how the image should appear.

# Load + GPU-upload model and texture (each happens once)
actor_model: Model3D = Assets.models.load("models/actor.gltf")
actor_tex:   Texture2D = Assets.textures.load("textures/actor.png", TextureSpec.smooth())

# The node wears both — no manual mesh/texture ids
actor: Node3D = Node3D.create("Actor")
actor.set_model(actor_model)
actor.set_texture(actor_tex)

Calling set_model and set_texture in either order is safe — the node binds the texture as soon as it has both. Each mesh keeps its own descriptor set, and meshes with no texture fall back to a 1×1 white pixel so geometry still renders.

Guide: Viewmodels (held items)

A first-person item should stay fixed relative to the camera, not the world. Rather than parenting it in the scene graph, render it by writing a raw model matrix built from the camera's world transform.

# Build: world = camera_world * local_offset, then push raw bytes
model: Mat4 = camera_world.mul(item_local)
buf: i64 = Memory.malloc_zeroed(64)
model.write_to_ptr(buf)
Engine.set_model_matrix_raw(item_mesh, buf)

Because set_model_matrix_raw bypasses the usual TRS pipeline, the item follows the camera exactly each frame and reads as a true held viewmodel.

Rig tip. A clean setup is a head node with the camera and the held item as its children, so look direction drives both together.

Guide: Performance Rules

Most frame-time problems come from doing setup work inside the update loop. Build resources once, reuse them, and only update transforms/game state per frame.

DoAvoidWhy
Load textures/models before the loop.Assets.textures.load(...) every frame.Loading reads files, decodes data, and uploads to GPU.
Create shared Material3D resources once.Rebuilding materials for thousands of objects per frame.Material changes may cause future descriptor/pipeline work.
Use position, rotation, translate, rotate for motion.Calling set_mesh repeatedly for animated objects.Changing mesh shape uploads/rebuilds geometry.
Reuse a mesh/material for many objects.Uploading the same model or texture many times.Shared resources reduce memory and prepare batching.
Use Engine.register_instance or game-owned batching for huge counts.Thousands of unique primitive uploads for identical objects.Instancing keeps draw/data cost lower.
Enable Engine.debug.render_stats(1) while measuring.Guessing based only on FPS.Frame time and render stats show where the cost is.
# Good: setup once
texture: Texture2D = Assets.textures.load("textures/crate.png", TextureSpec.pixel())
mat: Material3D = Material3D.create()
mat.set_texture(texture)

while Engine.running() == 1:
    dt: f32 = Engine.begin_frame()
    cube.rotate(0.0, 45.0 * dt, 0.0)
    Engine.render()
Rule of thumb. Setup code belongs before Engine.start() or before the first frame. Per-frame code should mostly update input, transforms, physics, AI, timers, visibility, and camera state.

Guide: Building & Optimization

Compile a game to a native executable with the Dolet compiler. An optimization level can be passed directly.

doletc game.dlt -o game.exe -O2      # or -O0 / -O1 / -O3
FlagEffect
-O0No optimization (default). Fastest builds, best for debugging.
-O1Basic optimization.
-O2Recommended for release builds. Strong speed gains, stable.
-O3Most aggressive optimization.

The level is forwarded straight to the C backend, so an unoptimized -O0 build and an -O2 build can differ dramatically in frame rate. Use -O2 when benchmarking or shipping.
